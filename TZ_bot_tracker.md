# Техническое задание: бот-трекер недельных задач бизнес-группы

> Документ предназначен для разработки в Cursor. Описывает архитектуру, модель данных, потоки и поэтапный план сборки. Стек выбран лёгким и async-нативным; код пишется на Python с разбором по шагам.

---

## 1. Контекст и цель

Бизнес-клуб; участники разбиты на небольшие группы, которые еженедельно встречаются, обсуждают прогресс и ставят задачи на следующую неделю. Встречи записываются и расшифровываются через Plaud.

**Важно для всего проекта:** это группа равных (peer-accountability), а **не** корпоративный контроль «сотрудник → руководитель». Тон бота — поддерживающий напарник, который помогает держать фокус, а не контролёр, который отчитывается начальству. Этот принцип определяет формулировки сообщений и логику приватности.

**Цель:** помочь участникам не терять фокус на недельных задачах и снять с ведущего ручную рутину сбора статусов.

---

## 2. Принципы продукта (определяют реализацию)

1. **Минимум усилий для занятого человека.** Чек-ин закрывается тапами по кнопкам и/или голосовым сообщением, а не печатанием. Текстовый/голосовой диалог с ИИ запускается только там, где он нужен.
2. **Приватность — две независимые оси:**
   - *Как задачи попадают в систему (вход):* авто-извлечение из транскрипта Plaud **или** приватный ввод напрямую боту (текст/голос).
   - *Кому уходит итог (видимость):* группе / только ведущему / только себе. Настраивается каждым участником.
   - Участник **всегда первым видит свой итог недели**, и только после его подтверждения сводка куда-либо уходит.
3. **Декомпозиция — по согласию, не навязанная.** При затыке бот *предлагает* помощь, а не заставляет дробить задачу.
4. **Подтверждение задач участником.** После извлечения из транскрипта участник проверяет и правит список — это и защита от ошибок атрибуции, и чувство собственности.

## 4. Архитектура

Один процесс на VPS:

- **Бот** (`aiogram`, long polling) слушает апдейты Telegram: команды, текст, голос, нажатия инлайн-кнопок.
- **Планировщик** (`APScheduler`) внутри того же event loop запускает недельные джобы: постановка задач (после встречи) и чек-ин (накануне следующей встречи), плюс опциональный пинг в середине недели.
- **Слой данных** (`SQLAlchemy`) хранит всё состояние: участники, недели, задачи, статусы, буфер диалога, сводки.
- **LLM-слой** вызывается из кода в двух местах: извлечение задач и трекинг-диалог.
- **Витрина Sheets** получает на запись только финальные сводки с учётом видимости.

Поток «от встречи до сводки» (карта экранов):

```
Онбординг (один раз)
        │
   ┌────┴────┐  выбор способа ввода
Авто из Plaud   Приватный ввод (текст/голос)
   └────┬────┘
 Подтверди задачи (правка и фиксация)
        │
 [Лёгкий пинг в середине недели — опционально]
        │
 Чек-ин накануне встречи (кнопки: сделал / в работе / затык)
        │  если «затык» и участник согласен
 Помощь при затыке (ИИ дробит на 2–4 шага → в план след. недели)
        │
 Свой итог недели (участник видит первым)
        │
   ┌────┴────┬─────────┐  выбор видимости
 Группе   Ведущему   Приватно
        │
 Прогресс /stats (серия, % выполнения за недели)
```

## 6. Потоки

### 6.1. Онбординг — `/start`
Бот связывает `telegram_chat_id` с записью `member` и за серию вопросов **кнопками** собирает настройки: способ ввода (`auto`/`private`), видимость, день и время чек-ина, нужен ли пинг. Это единственный длинный диалог, и он разовый. Если участника нет в базе — бот вежливо сообщает, что для него пока нет активной группы, и просит обратиться к ведущему.

### 6.2. Постановка задач (недельная джоба, после встречи)
- Для участников с `input_mode = auto`: взять `plaud_url` недели → получить транскрипт (§10) → вызвать LLM-извлечение (промпт 1, §9) с атрибуцией по `full_name` → создать `task` со `source = plaud`, `confirmed = false`.
- Для участников с `input_mode = private`: бот пишет в личку «Какие у тебя цели на эту неделю?»; участник отвечает текстом/голосом → LLM структурирует → создаёт `task` со `source = manual`.
- Оба варианта переходят к **подтверждению**: бот показывает список и инлайн-кнопки «Всё верно» / «Поправить». Правка — простым текстом или голосом, бот пересобирает список. После «Всё верно» — `confirmed = true`, `state = idle`.

### 6.3. Чек-ин (недельная джоба, накануне встречи)
Бот отправляет одно сообщение со всеми задачами недели; под каждой — инлайн-кнопки **«✅ Сделал / 🔄 В работе / ⛔ Затык»**. Тап обновляет `task.status` напрямую (без LLM) и редактирует сообщение (подсвечивает выбранный статус). Участник также может ответить голосовым/текстом — тогда подключается LLM-трекинг (§9, промпт 2), который распарсит ответ по задачам.

### 6.4. Обработка ответов
- **callback-хендлеры:** `callback_data` в формате `t:{task_id}:{status}` (компактно, в пределах 64 байт Telegram).
- **голос:** скачать файл → `whisper.cpp` → текст → дальше как текстовый ответ.
- **затык + согласие на помощь:** при статусе `stuck` бот спрашивает «Помочь разбить на шаги?» (кнопки Да/Нет). При «Да» — `state = decomposing`, LLM предлагает 2–4 конкретных шага (максимум один уточняющий вопрос для контекста). Участник правит/подтверждает → шаги сохраняются как `task` (`source = decomposed`, `week_id` = следующая неделя), исходная задача → `status = decomposed`.

### 6.5. Итог недели и маршрутизация по видимости
Когда статус всех задач определён и участник подтвердил завершение, LLM-трекинг выдаёт финальное сообщение с маркером `[REPORT_READY]`, после которого идёт резюме для ведущего. Бот:
1. отделяет резюме от диалоговой части;
2. показывает **участнику его сводку первым** и спрашивает кнопкой «Отправить по моим настройкам видимости?»;
3. после подтверждения сохраняет `summary` и маршрутизирует по `member.visibility`:
   - `group` → запись в Sheets-витрину, видимую группе (+ при желании пост в групповой чат);
   - `facilitator` → личное сообщение ведущему;
   - `private` → никуда не уходит, хранится только для участника.

### 6.6. Прогресс — `/stats`
По истории `task`/`summary` участнику показывается: серия закрытых недель, % выполненных задач, какие формулировки регулярно уходят в `stuck`/`decomposed`.

### 6.7. Пинг в середине недели (опционально)
Если `member.midweek_ping = true` — одно мягкое сообщение по одной невыполненной задаче, ответ в один тап. Не для всех, поэтому опция.

### Команды
`/start`, `/tasks` (текущий список), `/stats`, `/settings` (сменить видимость/время/пинг кнопками), `/help`.

## 8. LLM-слой (провайдер-агностичный + фолбэк)

**Никаких подписочных токенов в рантайме.** Бот ходит в платный API-ключ дешёвой мелкой модели, не обучающейся на данных (Gemini 2.5 Flash на платном Tier 1 / мелкая модель OpenAI / Claude Haiku). При текущих объёмах (сотни мелких вызовов в месяц) это единицы долларов.

Интерфейс:

```python
async def ask_llm(
    messages: list[dict],      # [{"role": "system"|"user"|"assistant", "content": ...}]
    *,
    json_mode: bool = False,
    timeout: float = 30.0,
) -> str: ...
```

- Провайдер выбирается из `.env` (`LLM_PROVIDER`, `LLM_MODEL`, `LLM_API_KEY`).
- **Фолбэк:** основной провайдер с таймаутом; при ошибке/таймауте — переключение на запасной (`LLM_FALLBACK_*`). Это лечит «залипания» отдельных моделей.
- Реализации провайдеров — за единым интерфейсом, чтобы добавление нового (в т.ч. OpenRouter как запасного) не трогало бизнес-логику.
- Для извлечения задач — `json_mode = true`, ожидаем JSON-массив строк.

## 9. Системные промпты (переписаны под принципы продукта)

### Промпт 1 — извлечение задач (с атрибуцией)
```
Ты — ассистент бизнес-группы, где участники на равных помогают друг другу держать фокус.
Перед тобой расшифровка еженедельной встречи группы. Выдели задачи, которые на этой
встрече взял на себя конкретный участник: {full_name}.

Правила:
- Бери только обязательства этого участника. Если по тексту нельзя уверенно понять,
  чьё это обязательство, — не приписывай его.
- Отбрасывай вводные слова, общие обсуждения и реплики не по делу.
- Формулируй каждую задачу как понятный результат: что должно быть сделано к концу недели.
- Если задач этого участника не нашлось — верни пустой массив, ничего не выдумывай.

Формат вывода: только JSON-массив строк с задачами, без приветствий и комментариев.
```

### Промпт 2 — трекинг и декомпозиция по согласию
```
Ты — поддерживающий напарник участника бизнес-группы (равный, не начальник).
Идёт чек-ин по итогам недели. Задачи участника: {tasks}.

Тон: тёплый, уважительный, без давления и менторства. Перед тобой занятой взрослый человек.

Правила:
1. Зафиксируй статус по каждой задаче: выполнено / в работе / не вышло. Прими как
   развёрнутый ответ, так и короткий — не переспрашивай лишнего, если всё уже сказано.
2. Если задача не вышла или застопорилась — НЕ дави и не читай нотаций. Спроси, хочет ли
   участник, чтобы ты помог разбить её на 2–4 конкретных шага. Только при согласии —
   предложи разбивку. При необходимости задай МАКСИМУМ один уточняющий вопрос для контекста.
   Шаги должны быть конкретными для его ситуации, а не общими ("изучить / спланировать / сделать").
3. Не превращай разговор в допрос.
4. Когда статус всех задач ясен и участник подтвердил, что закончил, выведи в самом конце
   маркер [REPORT_READY], а после него — короткое аналитическое резюме: что сделано,
   что застопорилось, какие шаги выделены на следующую неделю.
   Помни: это резюме первым увидит сам участник.
```

## 10. Интеграция с Plaud

Порядок предпочтения:
1. **Официальный экспорт/API Plaud**, если доступен по аккаунту — самый надёжный путь.
2. **Фолбэк — ручная вставка:** ведущий или участник присылает боту текст транскрипта (или бот принимает его из заранее заполненной колонки в Sheets). Это страховка, если автозабор недоступен.
3. **Скрейпинг публичной ссылки** — только если страница отдаёт текст; учесть возможный динамический рендеринг (SPA). Не делать единственным путём: при поломке вёрстки весь поток встаёт.

Любой сбой автозабора должен мягко переводить участника в режим ручной вставки, а не ломать джобу.

## 11. Витрина Google Sheets

- Доступ через service account (`gspread`).
- Пишутся **только финальные сводки** с учётом `visibility`: записи участников с `private` в групповой лист не попадают.
- Sheets — витрина на чтение; источник правды — БД. Бот не читает задачи из Sheets (кроме опционального резервного канала ручной вставки транскрипта, §10).

## 14. Деплой на VPS

- Виртуальное окружение, зависимости из `pyproject.toml`.
- `whisper.cpp` собирается отдельно; путь к бинарю и модели — в `.env`. Если ресурсов VPS мало — переключить `WHISPER_MODE=api`.
- `systemd`-юнит: автозапуск, рестарт при падении, логи в journald.
- Резервное копирование `data/app.db` (например, ежедневный дамп) — простой cron на уровне ОС.

## 15. План реализации по этапам (для сборки в Cursor)

Делаем инкрементально, каждый этап — рабочее состояние.

- **Этап 0 — каркас.** `pyproject`, `.env`, `config.py`, модели `member`/`group`, Alembic, бот отвечает на `/start`.
- **Этап 1 — онбординг и настройки.** Кнопочный онбординг, `/settings`, запись `member`.
- **Этап 2 — задачи (ручной режим).** `input_mode=private`: ввод целей боту, структурирование, экран подтверждения. (Сначала без LLM-извлечения — на простых правилах/ручном вводе, чтобы проверить поток.)
- **Этап 3 — чек-ин на кнопках.** Инлайн-клавиатуры, обновление `task.status`, редактирование сообщения.
- **Этап 4 — LLM-слой и авто-извлечение.** `ask_llm` + фолбэк, промпт 1, забор Plaud (с ручным фолбэком), `input_mode=auto`.
- **Этап 5 — декомпозиция и голос.** Опт-ин разбивка (промпт 2), `whisper.cpp`.
- **Этап 6 — итоги и витрина.** Маркер `[REPORT_READY]`, «сначала участнику», маршрутизация по видимости, запись в Sheets.
- **Этап 7 — прогресс и полировка.** `/stats`, midweek ping, деплой через systemd, бэкапы.

## 16. Критерии приёмки

- [ ] `/start` создаёт/находит участника и проводит онбординг кнопками; незнакомому участнику бот вежливо отказывает.
- [ ] В режиме `auto` задачи извлекаются из транскрипта с корректной атрибуцией конкретному участнику; при сбое автозабора — мягкий переход на ручную вставку.
- [ ] В режиме `private` участник вводит цели текстом/голосом, и они структурируются.
- [ ] Экран подтверждения позволяет принять или поправить список; до подтверждения задачи не считаются зафиксированными.
- [ ] Чек-ин закрывается тапами по кнопкам без печати; статусы корректно сохраняются и сообщение редактируется.
- [ ] При `stuck` бот предлагает помощь, и декомпозиция запускается **только при согласии**; шаги конкретны и переносятся в следующую неделю.
- [ ] Голосовой ответ расшифровывается и обрабатывается как текст.
- [ ] Финальную сводку участник видит **первым**; после подтверждения она уходит строго по его `visibility` (приватные — никуда).
- [ ] `/stats` показывает серию и % выполнения по неделям.
- [ ] LLM-фолбэк: при таймауте/ошибке основного провайдера запрос уходит к запасному.
- [ ] Состояние диалога переживает рестарт процесса (источник правды — БД).
- [ ] Смена `DATABASE_URL` на Postgres не требует правок в бизнес-логике.
