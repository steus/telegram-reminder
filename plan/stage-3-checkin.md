# Этап 3 — Чек-ин на кнопках + планировщик

> Сначала `plan/CONVENTIONS.md`, затем разделы ТЗ ниже.

## Цель

Ввести `APScheduler` и реализовать чек-ин на инлайн-кнопках: одно сообщение со
всеми задачами недели, под каждой — «✅ Сделал / 🔄 В работе / ⛔ Затык». Тап
обновляет `task.status` напрямую (без LLM) и редактирует сообщение. Планировщик
инициирует постановку задач и чек-ин по расписанию участника.

## Задачи

1. **`app/scheduler.py`**: `AsyncIOScheduler` с tz-aware джобами.
   - Джоба «постановка задач» (после встречи): для участников запускает поток сбора
     целей (пока — только `private`-ветка из этапа 2; `auto` придёт на этапе 4).
   - Джоба «чек-ин» (накануне встречи, по `checkin_weekday`/`checkin_time`/`timezone`
     участника): отправляет сообщение чек-ина.
   - Расписание per-member (у каждого свои день/время/tz). Решить: одна общая
     минутная джоба, которая проверяет, кому пора, — проще и переживает рестарт.
     (Зафиксируй выбор в коде комментарием-обоснованием в одну строку.)
   - Подключить старт планировщика в `app/main.py` (тот же event loop).
2. **Сервис** `app/services/checkin.py`:
   - собрать задачи недели участника, сформировать сообщение + клавиатуру
     (кнопки статусов под каждой задачей или сгруппированно — выбрать читаемый вид).
   - `apply_status(task_id, status)` через repo.
3. **Клавиатуры**: кнопки статусов с `callback_data = t:{task_id}:{status}`,
   где status ∈ {`done`,`in_progress`,`stuck`}. Следить за лимитом 64 байта.
4. **Роутер `checkin`** (`app/bot/routers/checkin.py`):
   - callback-хендлер парсит `t:{task_id}:{status}` → `apply_status` →
     `edit_message`, подсвечивая выбранный статус (✅/🔄/⛔ у нужной задачи).
   - Идемпотентность: повторный тап по тому же статусу не ломает.
   - Заглушка под «затык»: при `stuck` пока просто фиксируем статус (полноценная
     ветка декомпозиции — этап 5). Можно оставить TODO-хук.
5. Голос/LLM-разбор ответа в чек-ине пока **не** делаем (этапы 4–5).

## Definition of Done

- Планировщик стартует вместе с ботом, джобы видны в логах.
- Чек-ин-сообщение приходит по расписанию (и доступна тест-команда `/checkin_now`
  для ручного триггера в разработке).
- Тап по кнопке меняет `task.status` в БД и редактирует сообщение (подсветка).
- Формат `callback_data` соблюдён и укладывается в 64 байта.

## Локальная проверка

1. Выставить участнику ближайшее время чек-ина (или дёрнуть `/checkin_now`).
2. Получить сообщение с задачами и кнопками.
3. Потапать статусы → проверить изменение `task.status` в БД и редактирование сообщения.
4. Перезапустить контейнер → планировщик снова поднимается, задвоений нет.
