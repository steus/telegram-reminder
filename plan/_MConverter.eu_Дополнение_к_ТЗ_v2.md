**Дополнение к ТЗ (v2)**

*Онбординг, каскад целей год→квартал→неделя и анти-галлюцинационная архитектура*

Документ описывает изменения к исходному ТЗ «Автоматизированная система контроля и декомпозиции недельных задач на базе n8n, Plaud.ai, OpenAI и Telegram». Формат: сначала --- что добавлено, затем --- что меняется в существующих компонентах, затем --- обновлённый пайплайн ответа и честная граница гарантий.

Контекст пересмотра. Тестирование показало, что агент работает «вслепую»: он видит только список недельных задач, но не знает ни бизнеса участника, ни его цели. Продукт также сместился в сторону челлендж-групп, где участник сам ставит и выполняет задачи, сообщает, что не работает и что будет пробовать, а система должна анализировать это и подстраиваться --- при этом опираясь только на факты и не выдумывая.

**1. Что добавлено**

**1.1. Сценарий 0 --- Онбординг-опросник (JTBD)**

Новый сценарий, запускается при первом контакте участника (нет завершённого профиля по Telegram_Chat_ID). Агент по методологии Jobs To Be Done проводит интервью по блокам (Бизнес → Экономика → Каналы → Соцсети/трафик → Цель → Стиль) и на выходе формирует профиль в JSON + маркер \[ПРОФИЛЬ_ГОТОВ\]. Полный текст промпта --- в отдельном файле «Системный промпт: Онбординг-опросник».

**1.2. Каскад целей: год → квартал → неделя**

Продукт из недельного трекера становится инструментом достижения цели. Онбординг фиксирует годовой пункт назначения и квартальную цель; еженедельный цикл сшивает Plaud-задачи с этим каскадом. Смысл: каждую неделю система проверяет не «сделал ли задачи», а «приблизили ли задачи к квартальной цели».

**1.3. Таблица «Профиль_Пользователя»**

Постоянный контекст участника, отдельный от недельного буфера диалога. Ключ --- Telegram_Chat_ID (связь с таблицей «Задачи_Неделя»). Профиль хранится как JSON в одной ячейке --- чтобы чисто гонять его через LLM без десятков колонок.

| **Поле** | **Тип** | **Описание** |
|----|----|----|
| Telegram_Chat_ID | Строка/Число | Ключ, связь с «Задачи_Неделя» |
| Имя | Строка | Имя участника |
| Профиль_JSON | Длинный текст | Вся структура профиля (блоки business/economics/channels/social/goals/style) |
| Статус_Онбординга | Словарь | Не начат / В процессе / Завершён |
| Онбординг_Буфер | Длинный текст | История интервью (память на время онбординга) |
| Дата_Заполнения | Дата | Дата завершения онбординга |
| Дата_Обновления | Дата | Для квартального рефреша метрик и целей |

*Блок goals внутри Профиль_JSON (JTBD-ядро): year_goal, emotional_job, success_criteria, quarter_goal, quarter_deadline.*

**1.4. Таблица «Эксперименты» (лог гипотез для челлендж-групп)**

Челлендж-группа работает на цикле «гипотеза → попробовал → результат → следующая итерация». Без явного лога система не может анализировать прогресс без выдумывания. Эта таблица --- источник «внутренней правды» о пользователе, на которой стоит вся аналитика и подстройка.

| **Поле** | **Тип** | **Описание** |
|----|----|----|
| ID | Число | Идентификатор записи |
| Telegram_Chat_ID | Строка/Число | Связь с участником |
| Дата | Дата | Неделя фиксации |
| Гипотеза | Строка | Что участник пробует (напр. «рилсы 3×/нед для лидов») |
| Действие | Словарь | Сделал / Не сделал |
| Результат | Строка | Факт по итогу (напр. «10 рилсов, 2 заявки») |
| Вывод | Строка | Что понял (напр. «дорого по времени, конверсия низкая») |
| Следующая_итерация | Строка | Что пробует дальше |

**1.5. Анти-галлюцинационная архитектура (3 слоя)**

Цель переформулирована честно: не «ноль галлюцинаций» (недостижимо на свободной генерации), а «ноль непроверенных фактических утверждений, доходящих до пользователя». Достигается тем, что модель лишается роли источника фактов.

**Слой 1 --- Внутренняя правда (структурный ноль).**

- Всё о пользователе (цель, прогресс, «сколько раз пробовал X») --- не вспоминается моделью, а подставляется из таблиц; производные считаются в ноде Code, а не LLM. Галлюцинация здесь невозможна по построению.

**Слой 2 --- Внешняя правда (retrieval + extractive).**

- Любой внешний факт --- только через веб-поиск (Tavily). Для критичного факта модель цитирует спан источника дословно (extractive), а не пересказывает. Каждое утверждение несёт source_id на реально полученный документ. Нет источника → нет утверждения.

**Слой 3 --- Верификатор + детерминированный guard.**

- Второй LLM-проход (entailment): проверяет каждое утверждение против сниппетов-источников; неподтверждённое режется.

- Code-guard (не LLM): программно проверяет, что у каждого фактического поля есть валидный source_id. Несорсенное физически не выходит в Telegram.

- Абстенция как первоклассный путь: явно поощряется ответ «нет проверенных данных» --- это убирает выдумки «из вежливости».

**1.6. Tavily как Tool в ноде AI Agent**

GPT-4o через API сам в интернет не ходит --- ему нужен явный инструмент поиска. Внутри ноды AI Agent подключается Tavily (Add Tool → Tavily → API-ключ): агент использует веб-поиск как часть рассуждений, отдаёт сниппеты + URL. Опционально --- HTTP Request Tool для дочитывания полной страницы по URL. Фильтры Tavily по доменам и свежести (time range) держат источники доверенными и актуальными.

**2. Что меняется в существующих компонентах**

**2.1. Роутинг на входе (Telegram Trigger)**

Общий вход теперь маршрутизирует. Пришло сообщение → поиск профиля по Telegram_Chat_ID → если профиля нет или он не завершён → Сценарий 0 (онбординг); если завершён → обычный флоу (Сценарий 2), профиль подмешивается в контекст.

**2.2. Промпт Сценария 1 (экстракция задач)**

К системному промпту №1 добавляется подстановка профиля {Профиль_JSON}. Извлечённые из Plaud задачи фильтруются через нишу, стадию и приоритеты участника, а не формируются в вакууме.

**2.3. Промпт Сценария 2 (трекинг и декомпозиция)**

Ключевой пересмотр. К промпту №2 добавляется:

- Контекст участника {Профиль_JSON} --- советы и декомпозиция предметны для его бизнеса, а не общие.

- Каскад: агент постоянно связывает недельные задачи с {goals.quarter_goal} и {goals.year_goal}. Если задача не приближает к квартальной цели --- не приказывает, а подсвечивает вопросом («это работает на твою квартальную цель X или можно отложить?»). «Мягкое ведение».

- Опора на лог: агент ссылается на таблицу «Эксперименты» («ты уже дважды пробовал X, оба раза не окупилось --- сменим канал?»). Аналитика стоит на фактах пользователя.

- Обязательный поиск + цитаты: любой внешний факт --- только через Tavily, с source_id; при отсутствии источника --- честное «нет проверенных данных».

- Саммари для руководителя дополняется оценкой: насколько неделя продвинула участника к квартальной цели.

**2.4. Структура ответа агента**

Ответ формируется в структуре, где каждое фактическое утверждение помечено источником: {claim, source_id \| \"internal\" \| \"none\"}. Это делает возможными автоматическую верификацию и детерминированный guard.

**2.5. Новые ноды в воркфлоу**

- Нода-валидатор JSON после \[ПРОФИЛЬ_ГОТОВ\] (Сценарий 0): пустое обязательное поле → возврат агента на дозадавание.

- Нода-верификатор (LLM, Сценарий 2): entailment-проверка утверждений против источников.

- Code-guard (Сценарий 2): пропускает наружу только утверждения с валидным source_id или internal.

**3. Обновлённый пайплайн одного ответа (Сценарий 2)**

- 1\. Code / Excel --- собрать факты о пользователе (профиль, лог экспериментов, посчитанный прогресс) → в контекст как данные.

- 2\. AI Agent (Tools) с Tavily --- на внешние вопросы ищет, тянет спаны + URL.

- 3\. AI Agent генерирует ответ в структуре {claim, source_id}.

- 4\. LLM-верификатор --- entailment-проверка против сниппетов и внутренних данных.

- 5\. Code-guard --- режет любое непроверенное; только прошедшее уходит в Telegram.

*Цена: 2 LLM-вызова вместо одного + латентность. Для асинхронного Telegram-коуча это незаметно пользователю.*

**4. Что честно НЕ гарантируется**

«Полностью убрать галлюцинации» в буквальном смысле нельзя --- это фундаментальное свойство LLM, и клиенту так и сообщается. Достижимая и защитимая позиция:

- Правда о пользователе (его данные, прогресс) --- ноль по построению (инъекция + код).

- Правда о мире (рынок, тактики, цены) --- ноль непроверенных утверждений на выходе (retrieval + extractive + guard). Остаточный риск --- верификатор что-то пропустил, но всё сорсено и проверяемо пользователем.

- Свободная речь (тон, советы) --- держится в модальности «возможно, стоит попробовать», а не «это точно работает».

**Итог: наружу не выходит ни одного фактического утверждения без источника или внутренних данных за спиной. Остаточные ошибки становятся дешёвыми (сорсены, проверяемы), а не катастрофическими (голая уверенная выдумка).**
