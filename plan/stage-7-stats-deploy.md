# Этап 7 — Прогресс, полировка и деплой

> Сначала `plan/CONVENTIONS.md`, затем разделы ТЗ ниже.

## Цель

Закрыть продуктовые мелочи (`/stats`, midweek ping), отполировать и подготовить
прод-деплой на VPS через Docker Compose (основной путь) с systemd как
альтернативой, настроить бэкапы БД.

## Читать в ТЗ

- §6.6 (`/stats`), §6.7 (midweek ping), §6 «Команды», §14 (деплой, бэкапы),
  §16 (критерии приёмки — финальная сверка всех пунктов).

## Предусловия

- Этапы 0–6 готовы: полный функциональный поток работает локально.

## Задачи

1. **`/stats`** (§6.6): по истории `task`/`summary` — серия закрытых недель,
   % выполненных задач по неделям, какие формулировки регулярно уходят в
   `stuck`/`decomposed`. Логику вынести в `app/services/` (тестируемо).
2. **Midweek ping** (§6.7): если `member.midweek_ping=true` — джоба шлёт одно
   мягкое сообщение по одной невыполненной задаче, ответ в один тап.
   Встроить в планировщик (этап 3), отдельной джобой/проверкой.
3. **Полировка**: `/help` с актуальным списком команд (`/start`,`/tasks`,`/stats`,
   `/settings`,`/help`), единообразный тон, обработка краёв (нет недели/задач/группы).
4. **Прод-деплой (Docker, основной путь):**
   - финализировать `docker-compose.prod.yml`: `restart: always`, ресурсные лимиты,
     том `./data`, проброс `.env`, логи; whisper (том или `WHISPER_MODE=api`).
   - инструкция деплоя в `README.md`: clone → `.env` → `docker compose -f
     docker-compose.yml -f docker-compose.prod.yml up -d --build`.
5. **systemd (альтернатива без Docker, §14):** юнит-файл `deploy/bot-tracker.service`
   (venv, автозапуск, рестарт при падении, логи в journald) + краткая инструкция.
6. **Бэкапы** (§14): скрипт `scripts/backup_db.sh` (дамп/копия `data/app.db`) +
   пример cron-записи (ежедневно). Для Docker — бэкап тома.
7. **Финальная сверка по §16**: пройтись по чеклисту критериев приёмки, отметить
   галочки, зафиксировать незакрытые пункты в `PROGRESS.md`.

## Создаются/меняются файлы

`app/services/stats.py` (+ хендлер `/stats` в роутере), `app/scheduler.py`
(midweek), `app/bot/routers/common.py` (`/help`, `/stats`),
`docker-compose.prod.yml`, `deploy/bot-tracker.service`, `scripts/backup_db.sh`,
`README.md` (деплой + бэкапы).

## Definition of Done

- `/stats` показывает серию и % выполнения по неделям.
- Midweek ping приходит только тем, у кого включён, ответ в один тап.
- Прод поднимается на VPS через `docker compose ... up -d`; рестарт при падении.
- Бэкап `app.db` работает (проверен ручной прогон скрипта).
- Все критерии §16 пройдены либо явно отмечены как незакрытые с причиной.

## Локальная проверка / приёмка

1. Прогнать сквозной сценарий: онбординг → постановка задач → чек-ин (кнопки и
   голос) → затык+декомпозиция → итог → видимость → `/stats`.
2. Проверить рестарт-устойчивость состояния диалога (источник правды — БД).
3. Проверить, что смена `DATABASE_URL` на Postgres не требует правок логики
   (хотя бы поднять локальный Postgres и прогнать миграции + базовый поток).
4. Тестовый деплой на VPS: `docker compose -f ... -f docker-compose.prod.yml up -d`,
   проверить логи и автозапуск, прогнать бэкап-скрипт.
