# Этап 1 — Онбординг и настройки

> Сначала `plan/CONVENTIONS.md`, затем разделы ТЗ ниже.

## Цель

Кнопочный онбординг по `/start`: связать `telegram_chat_id` с `member`, собрать
настройки (способ ввода, видимость, день/время чек-ина, midweek-пинг). Команда
`/settings` для смены настроек. Незнакомому участнику — вежливый отказ.

## Читать в ТЗ

- §2 (принципы, особенно тон и приватность), §5 (`member`, `dialog_state`),
  §6.1 (онбординг), §6.7 (пинг), §7 (слой aiogram, FSM ↔ `dialog_state`),
  раздел «Команды» в §6.

## Предусловия

- Этап 0 готов: модели `group`/`member`, бот отвечает на `/start`, Docker/Alembic.

## Ключевое решение по «кого пускаем»

ТЗ: «Если участника нет в базе — бот вежливо сообщает, что активной группы нет».
Значит, участники заводятся не самим ботом из воздуха. Реализуй так:
- Ведущий заранее создаёт `group` и записи `member` (с `telegram_chat_id` или
  с привязкой по приглашению). На этом этапе достаточно: участник существует в БД,
  если его `telegram_chat_id` уже проставлен **или** есть механизм «pre-seed»
  (например, ведущий через временную админ-команду/прямую вставку добавляет
  участника по chat_id). Для локальной разработки опиши в README, как вставить
  участника вручную (seed-скрипт `scripts/seed_member.py` — допустимо).
- `/start` от неизвестного chat_id → вежливый отказ (тон напарника), без создания записи.

## Задачи

1. **`dialog_state`**: добавить модель из §5 + миграцию. Репозиторий-функции
   `get_or_create_dialog_state`, `set_state`, `update_context`.
2. **FSM** (`app/bot/states.py`): `StatesGroup` для онбординга/настроек. Синхронизация
   с `dialog_state.state` — источник правды БД (восстановление после рестарта).
3. **Роутер `onboarding`** (`app/bot/routers/onboarding.py`):
   - `/start`: найти `member` по chat_id. Нет — отказ. Есть, но не онбордился —
     запустить серию вопросов **инлайн-кнопками**:
     способ ввода (`auto`/`private`) → видимость (`group`/`facilitator`/`private`)
     → день чек-ина (0–6) → время (набор пресетов кнопками + возможность ввода) →
     midweek-пинг (да/нет). Сохранять выбор пошагово в `member`.
   - Уже онбордился — короткое приветствие + подсказка про `/settings`.
4. **Клавиатуры** (`app/bot/keyboards.py`): фабрики инлайн-клавиатур для каждого
   шага, компактные `callback_data`.
5. **Роутер `settings`** (`app/bot/routers/settings.py`): `/settings` — меню
   кнопок для смены видимости / дня / времени / пинга (переиспользует клавиатуры).
6. **Тон и формулировки**: все тексты — поддерживающий напарник, без канцелярита.
7. Подключить новые роутеры в `app/main.py`.

## Definition of Done

- `/start` от засиженного участника проводит весь онбординг кнопками; выбор
  сохраняется в `member`; повторный `/start` не ломает.
- `/start` от неизвестного chat_id → вежливый отказ, запись не создаётся.
- `/settings` меняет настройки и сохраняет их.
- После рестарта контейнера незавершённый онбординг продолжается с места (состояние из БД).

## Локальная проверка

1. `python scripts/seed_member.py` (или docker exec) — создать группу + участника
   со своим chat_id.
2. `/start` → пройти онбординг кнопками, проверить значения в БД.
3. Со «второго» (не засиженного) chat_id `/start` → отказ.
4. `/settings` → сменить видимость, проверить в БД.
5. Перезапустить контейнер на середине онбординга → состояние сохранилось.
