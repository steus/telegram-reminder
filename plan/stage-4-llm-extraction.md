# Этап 4 — LLM-слой и авто-извлечение из Plaud

> Сначала `plan/CONVENTIONS.md`, затем разделы ТЗ ниже.

## Цель

Реализовать провайдер-агностичный LLM-слой с фолбэком, промпт 1 (извлечение
задач с атрибуцией), интеграцию с Plaud (с мягким фолбэком на ручную вставку) и
поток `input_mode = auto`. Также подключить LLM-структурирование к `private`-вводу.

## Читать в ТЗ

- §8 (LLM-слой, интерфейс `ask_llm`, фолбэк), §9 (промпт 1), §10 (Plaud),
  §6.2 (постановка задач, ветка auto), §12 (ключи `.env`).

## Предусловия

- Этап 3 готов: планировщик, чек-ин, модели задач, экран подтверждения.

## Задачи

1. **LLM-клиент** (`app/llm/`):
   - `client.py`: `ask_llm(messages, *, json_mode=False, timeout=30.0) -> str`
     ровно по сигнатуре §8. Логика: основной провайдер с таймаутом → при
     ошибке/таймауте переключение на `LLM_FALLBACK_*`. Логировать факт фолбэка.
   - `providers/`: единый интерфейс `Provider` (async `complete(...)`).
     Реализации: `gemini.py`, `openai.py`, `anthropic.py`, `openrouter.py`.
     Минимум — провайдер, выбранный в `.env`; остальные можно как заглушки с
     понятным `NotImplemented`-сообщением, но интерфейс общий для всех.
   - `prompts.py`: промпт 1 и заготовка промпта 2 (дословно из §9, с `{full_name}`/`{tasks}`).
   - Конфиг расширить полями `LLM_*` и `LLM_FALLBACK_*` из §12.
   - `json_mode=True` для извлечения → ожидаем JSON-массив строк, безопасный парсинг.
2. **Plaud** (`app/services/plaud.py`), порядок предпочтения из §10:
   - (1) официальный экспорт/API — если настроен; (2) **ручная вставка** —
     ведущий/участник присылает текст транскрипта боту (или из колонки Sheets);
     (3) скрейпинг публичной ссылки — опционально, аккуратно с SPA.
   - Любой сбой автозабора **мягко** переводит в ручную вставку, не роняет джобу.
   - Минимально обязательная ветка для прохождения приёмки — ручная вставка +
     каркас под официальный API.
3. **Извлечение** (`app/services/extraction.py`): добавить
   `extract_tasks_from_transcript(transcript, full_name) -> list[str]` через
   `ask_llm` + промпт 1 (`json_mode`). Заменить/дополнить ручное структурирование
   `private`-ввода LLM-структурированием (точка расширения из этапа 2).
4. **Поток `auto`**: в джобе постановки задач (этап 3) для `input_mode=auto`:
   взять `week.plaud_url` → получить транскрипт → извлечь задачи с атрибуцией по
   `full_name` → создать `task` (`source=plaud`, `confirmed=false`) → отправить
   участнику **экран подтверждения** (переиспользуем из этапа 2).
5. Команда/механизм для ведущего: задать `plaud_url` недели и/или прислать
   транскрипт вручную (тест-команда допустима).

## Создаются/меняются файлы

`app/llm/{client,prompts}.py`, `app/llm/providers/{__init__,base,gemini,openai,anthropic,openrouter}.py`,
`app/services/{plaud,extraction}.py`, `app/scheduler.py` (ветка auto),
`app/config.py` (LLM-поля), `pyproject.toml` (httpx/SDK провайдеров), `.env.example`.

## Definition of Done

- `auto`: из транскрипта задачи извлекаются с корректной атрибуцией конкретному
  участнику; результат уходит на экран подтверждения (`confirmed=false`).
- Сбой автозабора Plaud → мягкий переход на ручную вставку, джоба не падает.
- `private`: ввод структурируется LLM (с разумным фолбэком на правила при сбое LLM).
- LLM-фолбэк: при таймауте/ошибке основного провайдера запрос уходит к запасному
  (проверяемо в логах/тестом с мок-провайдером).

## Локальная проверка

1. Прописать `LLM_PROVIDER`/`LLM_MODEL`/`LLM_API_KEY` (и фолбэк) в `.env`.
2. Юнит-тест с мок-провайдером: основной кидает таймаут → ответ от фолбэка.
3. Прислать тестовый транскрипт с именами 2 участников → проверить, что каждому
   приписаны только его задачи; чужие/общие реплики отброшены.
4. Сэмулировать сбой Plaud → бот предлагает прислать транскрипт вручную.
5. Проверить `private`-структурирование на сыром тексте.
