# Этап 8 — Профиль участника (анкета JTBD) и «умные» ответы

> Сначала `plan/CONVENTIONS.md`, затем этот файл. Первоисточники требований:
> `plan/_MConverter.eu_Дополнение_к_ТЗ_v2.md` и
> `plan/_MConverter.eu_Промпт_Онбординг_Опросник_JTBD.md`.
> **Важно:** документы написаны под n8n — у нас n8n НЕ используется. Всё переносим
> на нашу архитектуру: один Python-процесс, aiogram + SQLAlchemy async + `repo.py`,
> состояние диалога в `dialog_state` (§7).

## Зачем

Бот отвечает «вслепую»: видит только недельные задачи, не знает бизнеса и целей
участника. Добавляем постоянный **профиль участника** (JTBD-анкета) и подмешиваем
его в ответы, чтобы советы и декомпозиция были предметными, а не общими.

## Продуктовые решения (согласованы)

- **Анкета опциональна на входе.** После кнопочного онбординга — мягкое
  предложение заполнить, не обязаловка (упрощаем вход новичку).
- **Опросник — живое LLM-интервью** по JTBD-промпту (Сценарий 0), НО с
  **детерминированным прогрессом** («собрано X/8 ключевых пунктов · блок N/6») и
  возможностью **прервать и вернуться позже**.
- **Напоминание перед LLM — не блокирует, показывается каждый раз**, но только в
  **разговорных** сценариях (чек-ин/трекинг и декомпозиция). В «тихих» вызовах
  (извлечение из транскрипта, структурирование ручного ввода) не показываем.
- Команды: `/my_profile` (посмотреть), `/my_profile_fill` (заполнить/обновить/
  продолжить).

## Порядок работ (этапы)

Делаем **последовательно**. Этот файл — карта; на каждый под-этап запускается свой
прогон агента.

- **8A — Анкета + профиль + подмешивание в разговорные ответы** ← делаем сейчас.
- **8B — Каскад целей** год→квартал→неделя в промптах трекинга и саммари.
- **8C — Таблица «Эксперименты»** (лог гипотез челлендж-групп).
- **8D — Анти-галлюцинационный контур** (веб-поиск Tavily + LLM-верификатор +
  code-guard + структура ответа `{claim, source_id}`).

Ниже 8A расписан детально; 8B–8D — краткие рамки (детализируем перед запуском).

# 8A — Анкета JTBD, профиль, «умные» разговорные ответы

## Читать в первоисточниках

- `_MConverter.eu_Промпт_Онбординг_Опросник_JTBD.md` — весь (роль, принципы, 6
  блоков, обязательные поля, правила точности, JSON-схема, маркер `[ПРОФИЛЬ_ГОТОВ]`).
- `_MConverter.eu_Дополнение_к_ТЗ_v2.md` — §1.1 (Сценарий 0), §1.3 (таблица
  «Профиль_Пользователя»), §2.1 (роутинг на входе), §2.5 (нода-валидатор JSON).

## Предусловия

- Этапы 0–7 готовы (онбординг на кнопках, LLM-слой `ask_llm`, чек-ин/трекинг,
  декомпозиция, состояние диалога в `dialog_state.context_json`).

## Задачи

1. **Схема БД + миграция** (`app/db/models.py`, `alembic/`):
   - Таблица `member_profile` (ключ `member_id`, отдельно от `dialog_state`):
     - `member_id` — FK→member, PK, unique;
     - `status` — enum `OnboardingStatus` (`not_started`/`in_progress`/`completed`),
       хранится как VARCHAR (`native_enum=False`);
     - `profile_json` — Text, default `"{}"` (структура business/economics/channels/
       social/goals/style по JSON-схеме промпта);
     - `onboarding_buffer` — Text, default `"[]"` (окно истории интервью — для
       возобновления);
     - `progress_json` — Text, default `"{}"` (детерминированный прогресс:
       `{"block": 1..6, "filled": ["niche", ...]}`);
     - `filled_at` — DateTime null («Дата_Заполнения»);
     - `updated_at` — DateTime, `onupdate=func.now()` («Дата_Обновления», под
       квартальный рефреш в 8B).
   - Новый enum `OnboardingStatus`.
   - Новое значение `DialogStateEnum.onboarding_survey` (enum — VARCHAR, ALTER не
     нужен; миграция нужна только под новую таблицу).
   - `alembic revision --autogenerate -m "member profile (jtbd onboarding)"`.

2. **Репозиторий** (`app/db/repo.py`) — весь доступ к БД только здесь:
   - `get_or_create_profile(session, member_id)`, `get_profile(session, member_id)`;
   - `set_profile_status(session, member_id, status)`;
   - `save_profile_progress(session, member_id, buffer, progress_json)`;
   - `complete_profile(session, member_id, profile_json)` → status=completed,
     `filled_at=now`;
   - `reset_profile_for_refill(session, member_id)` (для `/my_profile_fill` поверх
     готового).

3. **Промпты** (`app/llm/prompts.py`):
   - `PROMPT_ONBOARDING_JTBD` — дословно из файла промпта, но **без упоминаний
     n8n**; сохранить принципы, 6 блоков, обязательные поля, правила точности,
     JSON-схему, маркер `[ПРОФИЛЬ_ГОТОВ]`. Добавить инструкцию выдавать **скрытый
     маркер прогресса** каждый ход: `[STATE]{"block": N, "filled": [...]}` (парсим
     и вырезаем, как уже делаем с `[STATUSES]`/`[REPORT_READY]`).
   - Хелпер `build_profile_context(profile_json) -> str` — компактный блок
     «Контекст участника» для подстановки в промпты. Если профиль пуст — короткая
     заметка «профиль не заполнен, не выдумывай факты о бизнесе».
   - Добавить плейсхолдер `{profile}` в `PROMPT_TRACKING` и `PROMPT_DECOMPOSE_STEPS`
     (разговорные сценарии). Инъекцию в извлечение/структурирование — в 8B.

4. **Сервис опросника** (`app/services/profile_onboarding.py` — новый; образец —
   `app/services/tracking.py`):
   - `start_survey(session, member)` → профиль в `in_progress`, буфер очищен,
     `DialogStateEnum.onboarding_survey`; вернуть приветствие + первый вопрос.
   - `process_survey_message(session, member, user_text)`:
     1. append в буфер; messages = `[system: PROMPT_ONBOARDING_JTBD]` + история;
     2. `ask_llm(...)`;
     3. распарсить `[STATE]` → обновить `progress_json`; распарсить
        `[ПРОФИЛЬ_ГОТОВ]`+JSON, если есть;
     4. **детерминированный guard** (аналог ноды-валидатора §2.5): проверить
        обязательные поля `niche, product, stage, bottleneck, year_goal,
        success_criteria, quarter_goal, hours_per_week`. Все на месте →
        `complete_profile`, state→idle, финал-подтверждение. Иначе профиль **НЕ**
        фиксируем — отдаём ответ LLM (он дозадаёт недостающее).
     5. Ответ пользователю = очищенный от маркеров текст LLM + строка прогресса,
        которую формируем МЫ из `progress_json` (напр. `Собрано ключевых пунктов:
        5/8 · блок 3 из 6. Напиши «пауза», чтобы прервать и вернуться позже.`).
   - `pause_survey(...)` — по слову «пауза»/кнопке: статус остаётся `in_progress`,
     буфер сохранён, state→idle, «Вернёшься командой /my_profile_fill».
   - Парсеры маркеров и валидатор обязательных полей — чистые функции (тестируемо).

5. **Роутер** (`app/bot/routers/profile.py` — новый; включить в `main.py` до `common`):
   - Хендлер сообщений при `state == onboarding_survey` (фильтр по dialog_state,
     как `InCheckin`); обрабатывает текст и «пауза».
   - `/my_profile` — показать профиль (форматируем из JSON) или «ещё не заполнен».
   - `/my_profile_fill` — старт/возобновление (если `in_progress` — продолжить с
     буфером; если `completed` — предложить обновить).
   - Callback `pf:start` (кнопка «Заполнить анкету») — запуск из любого места.

6. **Вход из онбординга** (`app/bot/routers/onboarding.py`):
   - В `_finish_onboarding` после сохранения настроек — **опциональное** мягкое
     предложение: кнопки `[Заполнить анкету (pf:start)] [Позже]`. «Позже» → idle,
     статус `not_started`. Профиль-строку создаём лениво (`get_or_create_profile`).
   - В `/start`: если статус `in_progress` — предложить продолжить анкету.

7. **Напоминание перед LLM** (не блокирует, каждый раз, только разговорные):
   - Хелпер `profile_nudge(profile)`: если статус ≠ completed → баннер «Чтобы
     отвечать точнее под твой бизнес — стоит заполнить анкету» + инлайн-кнопка
     `[Заполнить анкету]` (`pf:start`).
   - Встроить в `app/bot/routers/checkin.py` (`handle_checkin_freeform`) и в
     декомпозицию (`decompose`). Частоту вынести в одну константу — по умолчанию
     «каждый раз».
   - В эти же LLM-вызовы подмешивать профиль (`build_profile_context`).

8. **Команды/справка**:
   - `app/bot/command_names.py`: `CMD_MY_PROFILE`, `CMD_MY_PROFILE_FILL`.
   - `app/bot/commands.py`: пункты меню участника.
   - `app/bot/help_text.py`: описания.

9. **Тесты** (`tests/`): валидатор обязательных полей (guard незавершённого
   профиля), парсинг `[ПРОФИЛЬ_ГОТОВ]`+JSON и `[STATE]`, вырезание маркеров,
   вычисление прогресса (X/8, блок N/6), `build_profile_context` (пусто/полно).
   LLM/Telegram — мокать.

## Definition of Done

- Новый участник может пройти онбординг **без** анкеты (вход не усложнён).
- Анкета запускается из предложения после онбординга и по `/my_profile_fill`;
  показывает прогресс (X/8 · блок N/6); «пауза» прерывает, возврат восстанавливает
  диалог из буфера (переживает рестарт — состояние в БД).
- Профиль фиксируется как `completed` **только** когда собраны все обязательные
  поля (незаполненное не сохраняется как готовое — guard §2.5).
- В чек-ине и декомпозиции при незаполненном профиле показывается ненавязчивое
  предложение с кнопкой; при заполненном — ответы учитывают профиль.
- Миграция применяется; импорт/запуск без ошибок; линтер/типы чисты в изменённых
  файлах; `plan/PROGRESS.md` обновлён.

## Локальная проверка

1. `docker compose up` (или venv + `alembic upgrade head`), LLM-ключи в `.env`.
2. Новый участник → `/start` → пройти кнопочный онбординг → появляется
   предложение анкеты; «Позже» → бот работает как раньше.
3. `/my_profile_fill` → пройти интервью с мок/реальным LLM; проверить строку
   прогресса; на середине написать «пауза», затем `/my_profile_fill` — диалог
   продолжается с того же места.
4. Не назвать одно из обязательных полей → бот дозадаёт, профиль не фиксируется;
   после ответа — `[ПРОФИЛЬ_ГОТОВ]`, статус `completed`, `/my_profile` показывает
   данные.
5. Начать чек-ин с незаполненным профилем → видно предложение + кнопку; заполнить
   → повторный чек-ин учитывает контекст (проверяемо по промпту/логам).
6. Юнит-тесты зелёные.

# 8B — Каскад целей год → квартал → неделя (рамки)

- Подмешивать `goals.year_goal/quarter_goal/...` из профиля в `PROMPT_TRACKING`:
  агент связывает недельные задачи с квартальной целью, «мягкое ведение» вопросом
  («это работает на твою квартальную цель X или можно отложить?»), без приказов.
- Саммари дополнить оценкой: насколько неделя продвинула к квартальной цели.
- Инъекция профиля в извлечение задач и структурирование ручного ввода (§2.2).
- `updated_at`/квартальный рефреш метрик и целей.
- Первоисточник: доп.ТЗ §1.2, §2.2, §2.3.

# 8C — Таблица «Эксперименты» (рамки)

- Модель `experiment` (доп.ТЗ §1.4): `member_id, date, hypothesis, action(done/not),
  result, conclusion, next_iteration` + repo + миграция.
- Подстановка лога в трекинг («ты уже дважды пробовал X, оба раза не окупилось —
  сменим канал?»). Аналитика — на фактах пользователя, а не на выдумках.

# 8D — Анти-галлюцинационный контур (рамки, тяжёлый — отдельная проработка)

- Слой 1 (внутренняя правда): факты о пользователе — только инъекцией из таблиц,
  производные считаются в коде, не LLM.
- Слой 2 (внешняя правда): веб-поиск (Tavily как tool), extractive-цитаты со
  `source_id`; нет источника — нет утверждения.
- Слой 3: второй LLM-проход-верификатор (entailment) + детерминированный code-guard
  (наружу только `source_id`/`internal`); абстенция как первоклассный путь.
- Структура ответа `{claim, source_id | "internal" | "none"}`.
- Цена: +1 LLM-вызов и латентность — для асинхронного Telegram-коуча приемлемо.
- Первоисточник: доп.ТЗ §1.5, §1.6, §2.4, §3, §4.
